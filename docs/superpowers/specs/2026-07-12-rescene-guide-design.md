# 리센느 정밀 00초 가이드 웹앱 — 설계 문서

작성일: 2026-07-12

## 1. 배경 / 문제

러스챗(플러스챗) 리센느 방송 참여 신청은 **선착순 폼 제출**이다.
지금은 서버시간 페이지를 눈으로 보며 "59초 → 00초" 넘어가는 순간에 버튼을 누르지만:

- 내가 실제로 **00초에 눌렀는지, 59초에 눌렀는지 확인할 방법이 없다.**
- 사람 눈·손 편차 + 네트워크 지연 때문에 제출이 언제 접수되는지 모른다.

**목표:** 가장 정확하게 "00초에 접수되도록" 도와주는 도구를 만든다. 1단계는 **가이드(사람이 직접 클릭)** 방식.

## 2. 목표 & 비목표 (YAGNI)

**목표**
- 폰·PC 양쪽에서 쓰는 **설치 없는 웹페이지 하나**.
- 정확한 표준시각에 자동 동기화하여, **다음 00초까지 정밀 카운트다운**.
- **시각 + 소리** 신호로 "지금 눌러!" 순간을 알려줌.
- 네트워크 지연을 "미리 누르기(리드타임)"로 보정, 사용자가 조정 가능.

**비목표 (이번 단계에서 안 함)**
- 자동 클릭 / 자동 제출 (매크로) — 2단계로 분리.
- 폰 네이티브 앱 자동화.
- 로그인·캡차·폼 입력 자동화.

## 3. 측정 결과 (설계 근거)

2026-07-12, `artist.mnetplus.world` 리센느 페이지 대상 측정:

- 리센느 서버 시계 vs 표준시 = **약 +40~70 ms** (측정 불확실성 ±150ms).
- 즉 **서버는 사실상 표준시와 동일.** 사용자가 눈으로 본 "1초 차이"는 서버시간 페이지의 화면 새로고침 지연이었음.
- HTTP 응답 기반으로는 서버시간을 ±100ms보다 정밀히 읽을 수 없음(Date 헤더 1초 해상도 + 네트워크 지연).

**결론:** 표준시에 동기화 = 리센느 서버시간에 맞추는 것. **보정값 기본 0.**

## 4. 사용자 경험 (화면)

한 화면. 위→아래:

1. **큰 현재 시각** (서버/표준 기준, ms까지) + 동기화 상태 표시("동기화됨 ±20ms").
2. **목표까지 카운트다운** — 다음 00초, 또는 지정한 분(예: 21:00:00).
3. **마지막 3초 시각 신호** — 큰 숫자 3·2·1 + 원형 게이지가 0으로 수렴.
4. **정각 순간** — 화면 전체 초록 번쩍 + "삐!" 소리("지금!").
5. 마지막 3초 "똑·똑·똑" 틱 소리(리드타임만큼 앞당겨 울림).
6. **설정 영역** (접기 가능):
   - 미리 누르기(리드타임) ms — 기본 200, 슬라이더/입력.
   - 목표 시각 — "다음 00초" 또는 특정 시:분:초.
   - 수동 보정값 ms — 기본 0(보통 안 건드림).
   - 소리 on/off, 볼륨.

## 5. 구성요소

경계가 분명한 작은 단위로 나눔.

### (A) 시계 동기화 모듈 `clockSync`
- **하는 일:** 신뢰 가능한 시각 소스에 여러 번 왕복 측정, 최소 지연(min-RTT) 샘플로 `기기시계 오프셋`을 ms로 추정. 주기적 재동기화.
- **인터페이스:** `now()` → 추정된 참(true) 시각(ms). `status()` → {동기화여부, 추정오차}.
- **의존:** 시각 소스. 브라우저에서 NTP 불가 → HTTP 기반 시각 API(예: 공개 time API) 또는 응답 `Date` 헤더 + RTT 보정. 여러 소스로 견고화.
- **주의(구현 시 확인):** CORS. 브라우저에서 외부 시각 소스 읽기가 막힐 수 있음 → 읽히는 소스를 골라야 함. 막히면 (C)의 리드타임이 잔차 흡수.

### (B) 카운트다운 / 신호 모듈 `countdown`
- **하는 일:** `clockSync.now() + 보정값`을 기준으로 목표 시각까지 남은 시간 계산. `requestAnimationFrame`으로 매 프레임 갱신. 리드타임만큼 **앞당긴 시점**에 신호(시각+소리) 발생.
- **인터페이스:** `setTarget()`, `setLead(ms)`, `onTick(cb)`, `onGo(cb)`.
- **의존:** clockSync, 화면 렌더러, 오디오.

### (C) 설정 / 저장 모듈 `settings`
- **하는 일:** 리드타임·보정값·목표·소리설정 보관. `localStorage`에 저장(다음에 열면 유지). 폰/PC 환경별로 리드타임이 다를 수 있으니 프로필처럼 저장 가능하면 좋음(선택).
- **인터페이스:** `get()`, `set()`, 지속성.

### (D) 오디오 모듈 `beeper`
- **하는 일:** WebAudio로 틱음·정각음 생성(외부 파일 불필요, 지연 적음). 첫 사용자 상호작용 후 오디오 활성화(브라우저 자동재생 정책).

## 6. 데이터 흐름

```
시각 소스 ──RTT측정──> clockSync.now()(참 시각)
                              │  + settings.보정값
                              ▼
                         countdown  ──매 프레임──> 화면(숫자/게이지)
                              │  리드타임 앞당김
                              ▼
                     onGo ──> 초록 번쩍 + "삐!"  → 사용자가 클릭
```

## 7. 정확도 한계 & 보정 전략

- **표준시 동기화:** 자동, ms 단위 (min-RTT 방식). 확실히 정확.
- **서버 skew:** 측정상 표준시와 거의 동일 → 보정값 0. (수동 옵션만 유지)
- **네트워크 지연 + 사람 손 편차:** 하나의 **리드타임**으로 합쳐 경험적으로 조정.
  - 실전 몇 번 하며 "너무 늦음 → 리드타임↑, 너무 빠름(59초 접수) → 리드타임↓"로 자가 보정.
- **정직한 한계:** 사람이 클릭하는 가이드라 최종 정밀도 바닥은 **사람 손 ±30~50ms**. 밀리초 완벽이 필요하면 2단계(자동 제출)로.

## 8. 기술 스택

- 순수 **HTML + CSS + JavaScript** (프레임워크 없음, 백엔드 없음).
- 단일 페이지. 무료 정적 호스팅(예: GitHub Pages/Netlify)에 올려 URL로 폰·PC 공용,
  또는 PC에선 파일 더블클릭으로도 실행.
- 외부 시각 소스만 네트워크 사용. 그 외 오프라인 동작 가능.

## 9. 테스트

- `clockSync` 오프셋 계산: 가짜(mock) RTT 샘플 넣어 min-RTT 선택·오프셋 산출 검증.
- `countdown`: 가짜 시계 주입 → 목표·리드타임에 맞춰 onGo가 정확한 시각에 발화하는지.
- 신호 타이밍: 리드타임 X ms일 때 실제 목표보다 X ms 앞서 onGo 발생 확인.
- 수동 QA: 실제 브라우저(폰/PC)에서 time.is 옆에 두고 00초 일치 눈확인.

## 10. 향후 (2단계, 별도 spec)

- **웹 자동 제출:** 브라우저 자동화 또는 실제 제출 요청 재구성으로 00초 자동 발사.
- 매크로 탐지·약관 리스크 재검토 필요. 실전에서 가이드로 부족할 때만 진행.
